# ADR-0013：私有→公共升级口径收紧为「实质协作」+ 阈值 50

状态：已接受
日期：2026-06-14
关联：ADR-0007（页面模式与升级机制，本 ADR 取代其 §3 的口径与阈值）、ADR-0010（协作三件套）、ADR-0012（并发统一三方合并）

## 背景

ADR-0007 的自动升级口径是「非作者的（修订申请 + 可见评论含行内 + 主线直编修订）之和 ≥ 20」。
实践中这个口径把**轻量参与**（评论、未被采纳的修订申请）与**实质改动内容并被接纳**的协作混为一谈：
一篇文章靠评论刷数即可逼近阈值，而「转为公共页 = 让渡所有权、管理权移交板块编辑」是重决策，
不应被轻量互动驱动。门槛 20 对这一重决策也偏低。

## 决策

升级计数口径收紧为只数**实质协作**，阈值上调到 **50**：

- **计数 = 非作者的 ①「被采纳的修订申请」（`suggestions.status='merged'`）+ ②「直编落地的主线修订」之和。**
- ② 指 `revisions` 中 `suggestion_id is null`、`author_id` 非空且非作者署名的主线修订；
  并用 `suggestions.merged_revision_id` 排除「合入修订申请产生的 merge commit」，
  使一次被采纳的修订申请只由 ① 计一次，**不与 ② 重复计数**。
- **不再计入**：评论 / 行内评论、编辑建议（feedback），以及 open/under_review/changes_requested/rejected/
  outdated/withdrawn 等**未被采纳**的修订申请。
- 阈值默认 **50**，仍走 `site_settings['doc.publicize'].threshold` 覆盖（治理阈值不硬编码红线）。
- 检查点 `maybeAutoPromote` 改挂在「可能改变计数」的写路径后：**修订申请合入**（`mergeSuggestion`）与
  **他人直编修订落地**（`directEditPublished` 新增触发）；仍廉价、幂等、失败不连累主流程。
  评论写路径保留的调用退化为无害的冗余兜底（评论已不计数）。

升级的其余语义不变（ADR-0007）：CAS 防并发重复升级、写 `audit_log(doc.publicize)`、祝贺原作者、
保留 `owner_id`、手动升级走 `doc.set_visibility`。

## 取舍与后果

- 口径更贴合「公共价值 = 内容被多人实质改进并接纳」，杜绝评论刷量触发让渡所有权。
- 阈值 50 偏保守；冷启动期若希望共建更早发生，可经 site_settings 调低，无需改码。
- 旧数据自然兼容：历史评论不再计数，个别此前「靠评论接近 20」的私有页不会被动升级，符合预期。
- ADR-0007 §3 的口径与默认阈值以本 ADR 为准。
