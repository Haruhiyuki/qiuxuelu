# 0006. 任务队列用 pg-boss，推迟引入 Redis 系组件

- 状态：已采纳
- 日期：2026-06-10
- 关联：docs/02-architecture.md §1

## 背景

搜索索引同步、通知扇出、信任结算、锚点重映射都需要异步任务；但 1-3 人团队每多一个常驻服务都是真实运维负担。

## 决策

M0–M2 任务队列使用 pg-boss（队列落在 PostgreSQL，与业务同库同事务）；搜索同步走事务性 outbox 表。M3 实时协作上线时再引入 Valkey，且仅用于 presence/编辑租约等易失状态，不承载任何真相数据。

## 备选方案

- **Valkey + BullMQ 从第一天**（提案 A 原案）：功能更强，但冷启动期吞吐远未达到需要独立队列服务的量级，评审裁定推迟。
- **无队列，全部同步处理**：锚点重映射、通知扇出会拖慢提交事务，落选。

## 后果

- 正面：全栈常驻服务数最小（web + postgres，M1 加 worker + meilisearch）；任务与业务数据同事务，少一类一致性问题。
- 负面：PG 队列吞吐上限低于 Redis 队列——按里程碑量级评估完全够用；超出时迁移 BullMQ，接口已隔离在 worker 包。
