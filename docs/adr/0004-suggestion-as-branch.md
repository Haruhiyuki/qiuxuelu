# 0004. 编辑建议 = 真实修订分支，补丁是派生物

- 状态：已采纳
- 日期：2026-06-10
- 关联：ADR-0003、docs/02-architecture.md §3.3

## 背景

编辑建议需要：可讨论、可往返修改、可审计、主线前移后可检测冲突并变基。评审发现「存操作列表（ops）」方案在建议被要求修改后会整体重算 ops，**丢失建议自身的迭代史**，与「全历史可直观追溯」冲突。

## 决策

建议存为一条真实修订分支：`suggestions(base_revision_id, head_revision_id)`，head 沿 parent 链回到 base。补丁 = diff(base 树, head 树)，随时可重算。接受建议 = 三方块级合并产出 merge commit（author=建议人，committer=审校者）。冲突原子单位是块（段落）。

## 备选方案

- **ops 补丁表**（提案 B/C 共同方案）：表达紧凑，但往返迭代史丢失、与正式编辑走不同代码路径（双倍合并逻辑），落选。
- **Google Docs 式建议标记织入文档内容**：锚定永不漂移，但文档模型复杂化污染所有渲染视图，且与提交制历史冲突，落选。

## 后果

- 正面：建议与正式编辑共用全部内核代码（同一合并引擎服务并发变基/建议合并/协作 checkpoint 三条路径）；建议演化全程可追溯并喂给信任体系。
- 负面：建议会产生真实 revisions 行（标记为分支，列主线历史时过滤 `ref != draft/published` 的分支提交）；存储成本可接受。
