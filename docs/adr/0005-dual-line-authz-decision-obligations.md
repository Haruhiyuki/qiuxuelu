# 0005. 双线权限统一为「裁决 + 义务」的单一判定器

- 状态：已采纳
- 日期：2026-06-10
- 关联：docs/02-architecture.md §4

## 背景

权限有角色线（指派，带板块作用域）与信任线（TL0-TL4 晋升）两条来源；治理还要求「允许但进巡查队列」「允许但限速」这类带条件的许可。若鉴权返回布尔，治理逻辑会散落在各业务入口。

## 决策

`domain` 包提供唯一判定器 `can(actor, capability, resource): Decision`，返回裁决而非布尔：允许时附带义务（enqueue_patrol / rate_limit / pre_moderation…），拒绝时附结构化拒因（前端渲染为晋升引导文案）。判定顺序：制裁一票否决 → 角色线（作用域）→ 所有权特例 → 信任线（受 edit_policy 与红线约束）。角色专属红线能力（publish/protect/suspend/role.*/system.config）永不经信任线授予——晋升给能力，任命给权力。全部阈值入 site_settings 配置表。

## 备选方案

- **布尔 can() + 业务层各自处理巡查/限速**：治理语义碎片化，必然漏；落选。
- **CASL/RBAC 通用库**：表达不了「义务」与双线合成，落选；自研判定器为纯函数，单测成本低。
- **二维信任矩阵（按板块独立 TL）**：复杂度爆炸且不可解释，落选——信任全站一维，板块级权力走指派制。

## 后果

- 正面：治理机制（巡查、预审、限速、引导）全部收口一处；前后端共享判定器做按钮显隐与拒因解释。
- 负面：判定器载入 actor 快照（角色+TL+制裁）有查询成本——以请求级缓存对冲。
- 跟进：每个 Server Action 入口必须经 can()；CI 加静态检查防绕过。
