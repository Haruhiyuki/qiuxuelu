# 愿景与需求（PRD）

> 状态：v0.1 草案（2026-06-10）· 本文档是项目的「北极星」，所有架构与产品决策都应能回溯到这里。
> 修订方式：通过 PR 修改本文档，重大变更需在 `docs/adr/` 留下决策记录。
> 校准（2026-06-13）：§3 术语表、§4.2/4.3、§5 权限表已对齐已落地实现（ADR-0007 页面模式、0008 作者权、0009 AI 评论审核、0010 协作三件套改名、0011 编辑策略二元化）。权限细节以 `docs/03-permissions.md` 为活文档准绳。

## 1. 愿景

建设一个围绕**初中、高中、大学求学生涯**的多人协作开放博客平台：

- 任何亲历者都可以把自己的经验、心得、避坑指南沉淀为高质量文章；
- 内容不是一次性发表的「帖子」，而是像维基条目一样**可被社区持续修订、审校、完善的活文档**；
- 通过双线权限与信任等级机制，让协作既开放又有秩序；
- 作为长期运营的**公益项目**，内容对所有人免费开放，平台可自托管、可被复刻。

一句话定位：**博客的阅读体验 × Wiki 的协作深度 × 论坛的社区治理。**

## 2. 目标用户

| 画像 | 诉求 |
|------|------|
| 在读初/高中生 | 寻找学习方法、升学路径、心理调适经验；轻量参与（评论、纠错） |
| 大学生 | 检索专业选择、保研考研、实习就业经验；既是读者也是主力作者 |
| 毕业生/过来人 | 回馈式写作，沉淀长文经验 |
| 教师/家长 | 阅读为主，少量专业性补充 |
| 平台志愿者 | 板块管理、审批审校、社区治理 |

**特别注意**：核心读者包含未成年人。隐私保护（最小化收集个人信息）、内容安全（审批先行）、防引战与防营销号渗透是不可妥协的底线需求，而非附加项。

## 3. 核心概念术语表（全项目统一用语）

| 术语 | 英文/代码标识 | 定义 |
|------|--------------|------|
| 板块 | `section` | 顶层内容分区（如：初中 / 高中 / 大学 / 通用方法论），可分层 |
| 文章 | `document` | 一篇可协作的「活文档」，即用户所说的"博客" |
| 区块 | `block` | 文章的最小内容单元（段落、标题、代码、图片、提示框等），细粒度协作与追溯的基础 |
| 修订 | `revision` | 一次对文章内容的不可变快照记录（文档级），由若干块版本组成 |
| 修订申请 | `suggestion` | 对文章的修改提案 = 一条真实修订分支，需审校后合入（类比 PR / Google Docs 建议模式）。**ADR-0010 前称「编辑建议」** |
| 编辑建议 | `feedback` | **不改内容**，对全文/片段提的意见，送作者/编辑后台处理（ADR-0010 新增） |
| 行内评论 | `inline comment` | 锚定到具体区块乃至文字范围的评论 |
| 信任等级 | `trust level (TL)` | 自下而上的能力解锁等级（TL0–TL4） |
| 职务角色 | `staff role` | 自上而下指派的治理角色（超管/管理员/板块管理员/编辑） |
| 审批队列 | `review queue` | 待处理的新文章审批、建议审校、举报等工作项集合 |
| 贡献事件 | `contribution event` | 一切可计入信任度的行为记录（写作、建议被采纳、审校他人等） |

## 4. 功能需求

优先级标记：**P0** = 首个公开版本必须有；**P1** = 公开后第一阶段；**P2** = 远期。

### 4.1 用户与账号（P0）
- 邮箱注册/登录、会话管理、密码找回；OAuth（GitHub/微信等）为 P1。
- 公开资料页：昵称、头像、简介、教育阶段标签（自愿填写）、贡献统计、徽章（P1）。
- 隐私默认从严：不强制实名，不公开邮箱，未成年人相关字段最小化收集。
- 账号注销与数据导出（P1，但数据模型从第一天起预留）。

### 4.2 内容与协作
- **文章创建（P0）**：所有注册用户可新建文章草稿；提交发布 → 进入审批队列 → 板块管理员/编辑审批通过后公开。
- **块级编辑器（P0）**：所见即所得，支持段落、多级标题、列表、引用、代码块、图片、表格、提示框（callout）、分隔线；Markdown 快捷输入；草稿自动保存。
- **全历史追溯（P0）**：任何文章的任意历史修订可查看、可与任意另一修订做差异对比（块级 diff 可视化）；可一键回退（生成新修订而非删除历史）。
- **修订申请与编辑建议（P0，术语见 §3 与 ADR-0010）**：达到信任等级的用户可对已发布文章提交**修订申请**（`suggestion`，改内容、需审核合入，作者与有权者可接受/驳回/要求修改，过期与冲突有明确状态）；门槛更低的用户可提**编辑建议**（`feedback`，不改内容、只送意见，作者/编辑处理后回复）。
- **行内评论（P0）**：锚定到区块（P0）与文字范围（P1）；原文修改后锚点能重定位或优雅降级为「失锚」状态。
- **文章评论（P0）**：文末讨论区，支持回复层级（一层）。
- **协作编辑（P1）**：高信任用户/受邀协作者可直接编辑已发布文章（仍记入修订历史）；实时多人协同编辑为 P2。
- **多样贡献方式（P1+）**：纠错一键提交、补充参考链接、文章结构重组建议、翻译/转载规范化。

### 4.3 治理与权限
- **双线权限（P0）**：详见 §5。
- **审批队列（P0）**:新文章审批、建议审校，带工作项认领、通过/驳回理由、超时提醒（P1）。
- **举报与处置（P1）**：举报 → 队列 → 隐藏/警告/封禁，全程留痕。
- **审计日志（P0）**：一切权限敏感操作（审批、角色变更、删除、回退）可追溯。
- **内容保护（P1，已落地）**：管理员可对文章设「锁定编辑」——锁定后谁都不能直编，他人改走修订申请/编辑建议。早期设想的「半保护多档梯度」在页面模式（ADR-0007）接管后已简化为**锁定/开放二元**（ADR-0011）。

### 4.4 发现与阅读
- **阅读页（P0）**：一流中文排版（字号、行高、标点悬挂、段落节奏）、目录侧栏、阅读进度、暗色模式、移动端适配。
- **SEO（P0）**：SSR/SSG、语义化 HTML、OpenGraph、sitemap、RSS。
- **板块/首页（P0）**：板块导航、最新/精选文章流。
- **搜索（P1）**：对中文分词友好的全文搜索；P0 阶段可先提供标题/标签检索。
- **标签体系（P1）**、**合集/学习路径（P2）**。

### 4.5 通知与消息（P1）
- 站内通知：我的文章被评论/建议/审批结果、我的建议被处理、@提及。
- 邮件摘要可选订阅。数据模型 P0 预留（事件总线从第一天就存在）。

## 5. 双线权限模型（核心治理需求）

### 5.1 职务角色线（自上而下指派）

| 角色 | 作用域 | 核心能力 |
|------|--------|---------|
| 超级管理员 `superadmin` | 全局 | 一切；指派全局角色；系统配置 |
| 管理员 `admin` | 全局 | 用户管理、板块管理、指派板块角色、封禁、调信任等级、发公告 |
| 板块版主 `section_mod` | 指定板块 | 板块内评论管理、处置举报、下架、锁定编辑、精选、设为公共页 |
| 编辑 `editor` | 指定板块 | 板块内发布/审校、合并修订申请、回退、认领巡查、直接修订文章 |

### 5.2 信任等级线（自下而上晋升）

> 完整能力矩阵以活文档 `docs/03-permissions.md` 与 `packages/domain/src/authz/capabilities.ts` 为准；下表为概览，已并入 ADR-0008/0009/0010/0011 的调整。

| 等级 | 名称 | 解锁能力（叠加） | 晋升参考条件（系统自动 + 可人工调整） |
|------|------|----------------|----------------------------------|
| TL0 | 新成员 | 阅读、点赞/收藏、建文章/提审/文中发图（完整编辑能力）、举报、**发表评论**（AI 秒审，无预审/限速，ADR-0009） | 注册即得 |
| TL1 | 成员 | ＋行内批注、对**公共页**文章提编辑建议 | **发布首篇文章即 T1**（ADR-0010） |
| TL2 | 贡献者 | ＋对**公共页**提修订申请、对**私有页**提编辑建议 | 贡献积分（细则三期设计） |
| TL3 | 资深贡献者 | ＋在**公共页**直接修订他人文章（即时生效＋进巡查）、对**私有页**提修订申请 | 修订申请被采纳、举报命中率、滚动 100 天窗口考核（**可回落**） |
| TL4 | 共建者 | ＋处理公共页修订申请、管理修订 | TL3 长期保持 + 板块版主提名、管理员人工授予（无自动路径） |

规则要点：
- 两条线**取并集**生效：职务角色天然覆盖对应信任能力。
- 晋升由「贡献事件」流自动计算（可重放重算），TL3 按滚动窗口可自动回落，TL4 仅人工授予；人工可一票否决/破格提拔，全部留审计痕。
- 作者对**自己的**文章从 TL0 起即有完整协作权（直编 + 审核/合并他人修订申请，ADR-0008）。
- 所有阈值必须是**可配置参数**（`site_settings`）而非硬编码——社区规模变化时治理参数必然要调。

## 6. 非功能需求

- **可演进性**：数据模型（尤其修订/块/建议）一旦上线极难重做，必须第一版就正确；模块边界清晰，未来可拆服务。
- **可自托管**：单条 `docker compose up` 可起完整系统；不绑定专有云服务。
- **性能**：阅读页 P75 LCP < 2.5s（移动端 4G）；历史/对比页可接受稍慢但不可超 5s。
- **安全**：OWASP 基线；权限判定服务端强制，前端只做提示；速率限制；内容安全（XSS 防护是富文本系统的生命线）。
- **可观测**：结构化日志、错误追踪、关键业务指标（审批时长、建议采纳率）从早期就埋点。
- **国际化**：UI 文案走 i18n 框架（首发仅中文），内容模型不假设语言。
- **可访问性**：语义化 HTML、键盘可达、对比度达标（WCAG AA 为目标）。

## 7. 内容许可与法律（不可后补的前置决策）

- **内容许可**：协作编辑意味着作品被他人修改再发布，必须在注册与发文时明示许可协议。默认采用 **CC BY-SA 4.0**（同维基百科），署名归「原作者及贡献者列表」，修订历史即贡献凭证。
- **代码许可**：建议 AGPL-3.0（保证公益派生也开源），最终由项目所有者在首次公开发布前确认。
- **未成年人保护**：遵循最小化收集原则；UGC 审批先行本身即是内容安全机制；后续需补充社区公约与举报响应 SLA。

## 8. 明确不做（Non-goals，当前阶段）

- 不做私信/即时聊天（引导至评论区，降低治理负担）。
- 不做积分商城/任何变现机制。
- 不做算法推荐流（编辑精选 + 时间流足够，且符合公益定位）。
- 不做评论区无限层级嵌套（一层回复，保持讨论质量）。
